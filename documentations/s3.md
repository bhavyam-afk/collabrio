# S3 usage in Collabrio

This document describes which S3 components we use and three complete storage flows implemented in the codebase:

- Creator profile picture (onboarding)
- Brand logo (onboarding)
- Creator drafts (content uploads)

## S3 components and helpers used

- clients/s3.ts — creates an AWS SDK v3 `S3Client` configured from env vars:
  - `AWS_REGION`, `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`.
- clients/uploadToS3.ts — helper functions:
  - `uploadToS3(fileBuffer, key, contentType, publicRead = false)` — uses `PutObjectCommand` to upload and returns { url, key }.
  - `getPresignedUrl(key, expiresIn)` — uses `GetObjectCommand` + `getSignedUrl` to generate time-limited URLs for private objects.
- Low-level AWS SDK v3 primitives used: `S3Client`, `PutObjectCommand`, `GetObjectCommand` and `@aws-sdk/s3-request-presigner`.
- Environment variables required:
  - `AWS_S3_BUCKET` — bucket name
  - `AWS_REGION` — region for URL construction and client
  - `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY` — credentials

Note: `uploadToS3` exposes a `publicRead` argument but does not set ACLs; the comment in the file explains public access should be managed via bucket policies.

## Flow A — Creator profile picture (onboarding)

Files: `app/api/onboarding/creator/route.ts` → `uploadToS3`

Steps:
1. Frontend posts multipart form with `profilePic` file during onboarding.
2. Server validates MIME type against `allowedImageTypes = ["image/jpeg","image/png","image/webp"]` and size <= 10MB.
3. Server converts file to Buffer: `Buffer.from(await file.arrayBuffer())`.
4. Builds key: `onboarding/creators/${username}/profile-${Date.now()}`.
5. Calls `uploadToS3(buffer, key, file.type)` which:
	- Runs `PutObjectCommand` against `AWS_S3_BUCKET`.
	- Returns a stable object URL (constructed from bucket + region) and the stored key.
6. The returned `url` is saved to the DB as `creatorProfile.profilePicUrl`.

Result: Profile picture is stored in S3; frontend and other services read the `profilePicUrl` from the DB.

## Flow B — Brand logo (onboarding)

Files: `app/api/onboarding/brand/route.ts` → `uploadToS3`

Steps:
1. Frontend posts multipart form with `logo` file during brand onboarding.
2. Server validates MIME type (`allowedImageTypes`) and size <= 10MB.
3. Converts to Buffer and builds key: `onboarding/brands/${username}/logo-${Date.now()}`.
4. Calls `uploadToS3(buffer, key, file.type, true)` — note: the `publicRead` flag is passed true by the onboarding route but `uploadToS3` implementation does not set ACLs; public availability must be controlled by bucket policy.
5. The returned `url` is saved to the DB as `brandProfile.logoUrl`.

Result: Brand logo stored in S3 and referenced from `brandProfile.logoUrl`.

## Flow C — Creator drafts (content upload)

Files: `app/api/uploads/creatordraft/route.ts` → `uploadToS3`, `getPresignedUrl`

Purpose: creators upload draft content (videos/images) for a collaboration; content must be uploaded only after payment is locked in escrow.

Steps:
1. Creator sends a multipart/form-data POST with `collabId` and `file`.
2. Server authenticates session and loads `creatorProfile` and `collaboration` (includes `content` and `package` via Prisma).
3. Access checks:
	- Creator must own the collaboration.
	- `collab.collabStatus === "ACTIVE"`.
	- `PackageCollaboration.PaymentStatus` must be `BRAND_PAID` or `PLATFORM_HOLD` (ensures payment captured/escrowed).
	- A `BRAND_PAYMENT` transaction with `status = COMPLETED` is expected.
4. File validation:
	- Max size: 500MB
	- Allowed MIME types: `video/mp4`, `video/quicktime`, `image/jpeg`, `image/png`, `image/webp`.
5. Build deterministic S3 key:
	- `drafts/creators/${creator.id}/collabs/${collab.id}/${timestamp}-${safeFilename}` (safeFilename = sanitized original name).
6. Convert file to Buffer and call `uploadToS3(fileBuffer, fileKey, file.type)`.
7. `uploadToS3` returns `url` and `key`.
8. Generate a short-lived preview URL for the creator using `getPresignedUrl(s3Key, 60 * 60)` (1 hour) so private content can be previewed safely.
9. Update DB:
	- If `PackageCollaboration` doesn't exist → create one with `contentStatus = SUBMITTED`, `contentDraft = [{ url, key, type }]`, `draftSubmittedAt`.
	- If exists → append `{ url, key, type }` to `contentDraft`, set `contentStatus = SUBMITTED`, update `draftSubmittedAt`.
10. Response includes `fileUrl`, `previewUrl`, uploadedCount and updated collaboration metadata.

Data shape stored in DB for drafts (as used in code):
```
contentDraft: [ { url: string, key: string, type: 'video'|'image' }, ... ]
```

Result: Draft stored in S3 (private by default). The DB holds the object URL and key; UI uses `getPresignedUrl` to preview private objects.
